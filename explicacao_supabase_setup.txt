(escrito por mim, Adriano, para eu conseguir lembrar-me disto depois)

OBJETIVO GERAL
------------------------------------------------------------------------
Este ficheiro é um "guião" de instruções que eu colo de uma vez no SQL
Editor do Supabase. Quando o corro, ele constrói do zero a base de dados
inteira da minha loja: cria as "gavetas" (tabelas) onde vão ficar os
produtos, os clientes e as encomendas, tranca essas gavetas para que só
eu (admin) possa mexer no que não deve ser público, e já mete lá dentro
os meus 12 produtos atuais, para eu não ter de os escrever à mão outra
vez no site.

Cada bloco do ficheiro faz uma coisa diferente. Vou explicar por ordem,
tal como aparece no ficheiro.

------------------------------------------------------------------------
BLOCO 1 — EXTENSÕES
------------------------------------------------------------------------
  create extension if not exists "pgcrypto";

O Postgres (o "motor" de base de dados que o Supabase usa por baixo) não
vem com tudo ligado por defeito. Esta linha liga uma funcionalidade
chamada "pgcrypto", que é o que permite gerar automaticamente aqueles
códigos compridos e únicos (chamados "uuid") que uso como identificador
de cada produto, cada cliente, cada encomenda, etc. Sem isto, o
gen_random_uuid() que uso mais abaixo não funcionava.

------------------------------------------------------------------------
BLOCO 2 — TABELAS (as "gavetas" onde os dados ficam guardados)
------------------------------------------------------------------------
Criei 7 tabelas. Cada "create table" define os campos (colunas) que essa
gaveta vai ter e as regras de cada campo.

  2.1 categories
      Guarda as categorias da loja: corrida, treino, futebol, estilo.
      - id: código único gerado automaticamente.
      - slug: o nome "técnico" (ex: 'corrida'), tem de ser único.
      - label: o nome bonito que aparece no site (ex: 'Corrida').

  2.2 products
      A tabela mais importante — um produto da loja.
      - category_id: liga o produto a UMA categoria da tabela anterior
        (isto chama-se "chave estrangeira" — garante que nunca crio um
        produto com uma categoria que não existe).
      - price / original_price: preço atual e preço "antes do desconto".
        Pus uma regra (check) que obriga original_price a ser sempre
        maior que price, para não haver enganos.
      - sizes: uma lista de tamanhos (ex: 38, 39, 40...).
      - active: em vez de apagar um produto a sério quando o elimino no
        admin, esta coluna passa a "false" — assim guardo o histórico
        (uma encomenda antiga continua a mostrar o nome do produto
        mesmo que ele já não esteja à venda).
      Também criei dois "índices" (idx_products_category e
      idx_products_active) — são atalhos que tornam as pesquisas na
      tabela mais rápidas quando a loja tiver muitos produtos.

      Por fim criei uma função (set_updated_at) e um "trigger"
      (trg_products_updated_at). Um trigger é uma regra automática: "de
      cada vez que um produto for atualizado, atualiza também a data
      updated_at sozinho". Nunca preciso de me lembrar de fazer isso à
      mão.

  2.3 customers
      Os dados de quem compra (nome, email, morada). Como a loja não tem
      contas de cliente, cria-se um registo novo automaticamente em cada
      compra.

  2.4 orders
      Uma encomenda. Liga-se a um customer (customer_id) e guarda
      subtotal, portes (shipping) e total. Tem uma coluna "status" com
      uma lista fechada de valores possíveis (pendente, pago, enviado,
      entregue, cancelado) — assim nunca escrevo um estado inválido por
      engano.

      Criei também uma sequência (order_ref_seq) e outro trigger
      (trg_orders_reference) que gera sozinho a referência da encomenda
      no formato "AC000001", "AC000002", etc., cada vez que uma
      encomenda nova é criada. Não preciso de inventar esse número no
      site.

  2.5 order_items
      Os artigos dentro de uma encomenda (um produto + tamanho +
      quantidade). Guardo o "product_name" como cópia do nome do
      produto na altura da compra — assim, mesmo que eu apague ou mude
      o nome do produto mais tarde, a encomenda antiga continua a
      mostrar o nome certo.

  2.6 newsletter_subscribers
      Só os emails de quem se inscreve na newsletter no site.

  2.7 job_applications
      Candidaturas de emprego da página "Carreiras", caso um dia ligue
      esse formulário à base de dados a sério.

------------------------------------------------------------------------
BLOCO 3 — ROW LEVEL SECURITY (RLS) — o "cadeado" de cada gaveta
------------------------------------------------------------------------
Isto é a parte mais importante para a segurança da loja.

O meu site é 100% feito em HTML/JS que corre no browser de quem visita
a loja. Isso significa que a "chave" que uso para ligar ao Supabase
(a anon key) fica visível para qualquer pessoa que abra o código da
página. Se eu não tivesse mais nada a proteger a base de dados,
qualquer pessoa curiosa podia usar essa chave para ler ou até apagar
tudo diretamente, sem passar pelo meu site.

O RLS resolve isto: primeiro "tranco" TODAS as tabelas com
"alter table ... enable row level security" (sem isto, mais ninguém
consegue ler nem escrever nada, nem eu). Depois, para cada tabela,
escrevo "políticas" (create policy) que dizem exatamente o que é
permitido e a quem:

  - categorias e produtos: QUALQUER pessoa pode LER (para a loja
    aparecer a toda a gente), mas só um utilizador "authenticated"
    (ou seja, com sessão de login válida — o admin) pode criar, editar
    ou apagar.
  - clientes e encomendas: qualquer pessoa pode CRIAR (para o checkout
    funcionar mesmo sem login de cliente), mas só o admin consegue LER
    a lista — assim um estranho não consegue ver moradas e emails de
    outras pessoas.
  - newsletter e candidaturas: mesma lógica — qualquer um escreve
    (subscreve/candidata-se), só o admin lê a lista.

Resumindo: o RLS é o que faz o site funcionar como loja pública, sem
deixar ninguém bisbilhotar ou estragar dados que não são dele.

------------------------------------------------------------------------
BLOCO 4 — SEED DAS CATEGORIAS
------------------------------------------------------------------------
  insert into categories (slug, label) values (...);

"Seed" quer dizer "semente" — são os dados iniciais que meto logo na
tabela para o site não começar vazio. Aqui insiro as 4 categorias que já
usava no site antigo.

------------------------------------------------------------------------
BLOCO 5 — SEED DOS PRODUTOS
------------------------------------------------------------------------
  insert into products (...) values (...);

O mesmo princípio, mas para os 12 produtos que já tinha no site (antes
guardados no ficheiro js/db.js, dentro do browser). Uso
"(select id from categories where slug='corrida')" em vez de escrever o
código da categoria à mão — isto vai automaticamente buscar o id certo
da categoria "corrida" que criei no bloco anterior, para ligar o produto
à categoria correta.

RESUMO EM UMA FRASE
------------------------------------------------------------------------
Este ficheiro constrói a "casa" toda da minha base de dados (as
divisões/tabelas), põe fechaduras nas portas certas (RLS), e já mobila
a casa com os móveis que já tinha (as 4 categorias e os 12 produtos).
